## 0. Purpose

TP01 is the daily-use anchor extraction workflow for turning one or more reference images into a reusable Anchor Set.

This file is the core workflow only. It defines source authority, execution order, stop points, runtime contracts, validation, manifest, package governance, and failure handling.

Concrete generation prompts for AnchorA / AnchorB / AnchorC live in:

```text
TP01_PROMPTS.md
```

This split is intentional:

```text
Tp01.md = stable protocol / operating system
TP01_PROMPTS.md = editable extraction prompt module
```

When prompt behavior needs refinement, update `TP01_PROMPTS.md` first. Do not rewrite core workflow unless the protocol itself changes.

```text
AnchorC.PNG
https://raw.githubusercontent.com/hoiyu915-droid/Tp01/main/AnchorC.PNG
```

GitHub connector is used for:

```text
- repository metadata
- file SHA
- TP01_SOURCE_LOCK.md text
- Tp01.md text
- TP01_PROMPTS.md text
- text/procedure audit
```

Raw GitHub links are used for:

```text
- AnchorA.PNG visual content
- AnchorB.PNG visual content
- AnchorC.PNG visual content
```

Important distinction:

```text
File existence / SHA / metadata is not visual read.
An anchor may be called "read" only after the actual image content has been opened or visually inspected.
```

Canonical anchor files define role grammar only. They are not current-run content.

```text
AnchorA.PNG baseline = what a layout grammar anchor is
AnchorB.PNG baseline = what a style DNA anchor is
AnchorC.PNG baseline = what an asset / object vocabulary anchor is
```

The current run must be derived from the current user source inputs through runtime files before image generation.

## 4. Invocation Contract

When the user says `tp01` and does not attach a TP01 file:

1. Fetch repository metadata from `hoiyu915-droid/Tp01`.
2. Read `TP01_SOURCE_LOCK.md`.
3. Read `Tp01.md`, including SHA / metadata.
4. Read `TP01_PROMPTS.md`, including SHA / metadata.
5. Read AnchorA / AnchorB / AnchorC from the canonical raw links.
6. Verify all source components are available:
   - `Tp01.md`
   - `TP01_PROMPTS.md`
   - `AnchorA.PNG`
   - `AnchorB.PNG`
   - `AnchorC.PNG`
7. **STOP POINT 1 — Anchor Read Report**
   - Before any runtime or generation, output a report confirming whether AnchorA / B / C were actually read as visual content.
   - This report must distinguish:
     - file existence / metadata
     - SHA availability
     - actual visual-read success
     - unresolved failures
   - If any anchor image was not actually read visually, stop the TP01 run here.
8. Inspect the current user reference / task only after Stop Point 1 passes.
9. Generate current-run runtime files from the current source inputs:
   - `AnchorA.runtime.md`
   - `AnchorB.runtime.md`
   - `AnchorC.runtime.md`
10. **STOP POINT 1R — Runtime Report**
    - Evaluate whether each runtime was derived from the current source input.
    - Confirm whether:
      - AnchorA runtime is layout grammar only
      - AnchorB runtime is style DNA only
      - AnchorC runtime is asset / object vocabulary only
      - the three runtimes are sufficiently separated
      - no runtime silently reverted to canonical baseline content
    - Do not generate anchor images before this report passes.
11. Create `SOURCE_DIR / SOURCE_INDEX.md` draft if creating a package.
12. Generate `AnchorA.PNG` sequentially from `AnchorA.runtime.md` and `TP01_PROMPTS.md` AnchorA prompt rules.
13. Validate `AnchorA.PNG`.
14. Generate `AnchorB.PNG` sequentially from `AnchorB.runtime.md` and `TP01_PROMPTS.md` AnchorB prompt rules.
15. Validate `AnchorB.PNG`.
16. Generate `AnchorC.PNG` sequentially from `AnchorC.runtime.md` and `TP01_PROMPTS.md` AnchorC prompt rules.
17. Validate `AnchorC.PNG`.
18. **STOP POINT 2 — Generated Anchor Report**
    - After generating A / B / C, output a report evaluating:
      - whether AnchorA is layout grammar only
      - whether AnchorB is style DNA only
      - whether AnchorC is asset / object vocabulary only
      - whether the three anchors are sufficiently separated
      - whether any anchor drifted into a finished card, moodboard, design guide, or full story scene
      - whether each anchor follows its runtime and prompt module
    - Do not write `manifest.md`, finalize `SOURCE_INDEX.md`, create a zip, or upload before this report passes.
19. Finalize `SOURCE_DIR/SOURCE_INDEX.md` only after Stop Point 2 passes.
20. Write `manifest.md` only after Stop Point 2 passes.
21. **STOP POINT 3 — Package Preflight Report**
    - Before building the final seed package / zip, output a report confirming:
      - all required files exist
      - source references are recorded
      - runtime files are recorded
      - prompt module version is recorded
      - manifest is aligned with A / B / C and runtime
      - no overview image replaced A / B / C
      - package is complete and ready
22. Generate zip / upload only after Stop Point 3 passes and only if the user requested a package / zip / upload.
23. Report any failed source explicitly.
24. Do not substitute GDrive unless the user explicitly asks for it.

Hard rule:

```text
Do not collapse TP01 into a single uninterrupted generate-and-package run.
Do not generate anchors before AnchorA/B/C runtime exists.
Do not generate anchors without reading TP01_PROMPTS.md.
The stop points and runtime gate are mandatory execution gates.
```

## 6. Anchor Separation Model

### AnchorA.PNG — layout grammar only

Purpose:

```text
Extract page skeleton, reading order, spacing, panels, card hierarchy, and content zones.
For non-card references, extract the source image's spatial grammar: frame ratio, viewpoint, axes, depth layers, node positions, negative space, flow, repetition, and structural rhythm.
```

Must include:

```text
- page ratio or frame ratio
- viewpoint
- primary structure
- depth layers
- dominant axes
- major nodes
- negative space map
- reading / viewing flow
- repetition or rhythm
- structural relationships
```

Must not include:

```text
- complete character
- complete object inventory
- readable final text
- original topic content
- finished illustration
- brand / identity carryover
- style DNA board
- canonical baseline layout copied as the current-run content
```

Failure condition:

```text
If AnchorA looks like a finished infographic, annotated poster, or source-photo explanation board rather than a reusable skeleton, it fails.
If AnchorA reflects the canonical baseline instead of the current source input, it fails.
```

### AnchorB.PNG — style DNA only

Purpose:

```text
Extract visual language without fixing the final layout.
```

Must include:

```text
- material impression
- texture language
- line quality
- edge behavior
- palette family
- contrast profile
- light and shadow mood
- surface character
- overall visual tone
```

Must not include:

```text
- fixed full layout
- complete infographic content
- original story
- object inventory
- workflow logic
- topic conclusion
- moodboard / design guide / brand manual behavior unless explicitly requested
- canonical baseline style copied as the current-run content
```

Failure condition:

```text
If AnchorB becomes a full information card, style guide poster, moodboard with headings, or full scene reconstruction, it fails.
If AnchorB reflects the canonical baseline instead of the current source input, it fails.
```

### AnchorC.PNG — asset / object vocabulary only

Purpose:

```text
Extract reusable visual components and object vocabulary.
```

Must include when present in the source:

```text
- poses
- expressions
- gestures
- props
- scene elements
- icons
- arrows
- labels only when essential and minimal
- cards
- reusable UI / infographic components
- primary objects
- secondary objects
- connectors or relationships
- micro details
```

Must not include:

```text
- complete page composition
- complete story
- full hierarchy
- single fixed conclusion
- large unified scene reconstruction
- layout-guide behavior
- style-guide behavior
- brand-specific or identity-specific objects copied from the source
- canonical baseline assets copied as the current-run content
```

Failure condition:

```text
If AnchorC reads like a finished story scene instead of a reusable asset sheet, it fails.
If AnchorC reflects the canonical baseline instead of the current source input, it fails.
```

## 7. Runtime Contract

Before generating `AnchorA.PNG`, `AnchorB.PNG`, or `AnchorC.PNG`, TP01 must generate three runtime files derived from the current source inputs.

Required runtime outputs:

```text
AnchorA.runtime.md
AnchorB.runtime.md
AnchorC.runtime.md
```

These runtime files are mandatory intermediate artifacts.

Purpose:

```text
- convert current reference inputs into explicit extraction rules
- prevent silent fallback to canonical baseline content
- separate source interpretation from image generation
- make A / B / C generation auditable
```

Hard rule:

```text
Do not generate AnchorA.PNG before AnchorA.runtime.md exists.
Do not generate AnchorB.PNG before AnchorB.runtime.md exists.
Do not generate AnchorC.PNG before AnchorC.runtime.md exists.
Canonical anchor files define role grammar only; they do not substitute for current-run runtime.
```

## 11. Default Style Policy

Unless the user explicitly gives another style, new TP01 anchor packages should use:

```text
- portrait / vertical layout by default
- soft Japanese hand-drawn card style
- clean readable structure
- low-saturation palette
- fine linework
- white or pale paper background
- rounded modules only when relevant
- social-card readability only when target is a card
```

Default transformation rule:

```text
copy structure, not identity
copy workflow logic, not brand
copy visual function, not exact artwork
copy mood category, not artist-specific style
copy current source extraction through runtime, not canonical baseline content
```

## 14. Failure Handling

If any required GitHub source fails:

```text
- Say exactly which file failed.
- Distinguish metadata success from visual-read success.
- Do not claim an anchor was "read" if only SHA / existence was confirmed.
- Do not infer anchor visual content from Tp01.md text alone.
- Do not infer prompt module content from memory.
- Do not infer anchor visual content from memory, prior conversation, or previous seed packages.
- Do not silently search GDrive.
- Ask for explicit override only if the user wants to continue with another source.
- Stop before runtime generation if Stop Point 1 cannot pass.
```

If runtime generation fails:

```text
- Say exactly which runtime failed.
- Distinguish source-read success from runtime-derivation success.
- Do not generate the corresponding anchor image.
- Do not infer runtime from canonical anchor role definitions alone.
- Do not silently reuse old runtime, old package, or old output.
- Stop before AnchorA / B / C image generation if Stop Point 1R cannot pass.
```

If generated anchors fail separation rules:

```text
- Stop after Stop Point 2.
- Report which anchor failed and why.
- Regenerate the failed anchor only, using the corresponding runtime and TP01_PROMPTS.md rules.
- Do not write manifest or build package until AnchorA / B / C all pass.
```

If package assembly fails:

```text
- Stop at Stop Point 3.
- Report missing or inconsistent files.
- Do not zip incomplete outputs.
```

Common failures and corrections:

```text
Tp01.md still mentions Drive as default -> rewrite procedure to GitHub canonical source.
TP01_PROMPTS.md missing -> create or restore prompt module before generation.
runtime copied canonical baseline instead of current source -> regenerate runtime from current REF set.
AnchorA.runtime contains style or object inventory -> reduce to layout grammar only.
AnchorB.runtime contains fixed page skeleton -> remove composition lock.
AnchorC.runtime contains full story -> split into reusable object vocabulary.
runtime missing must_capture / must_avoid / generation_rule -> complete the runtime schema.
runtime not created but anchor generated -> invalid run; discard anchor and regenerate from runtime stage.
AnchorA too finished -> redo as layout skeleton.
AnchorB includes moodboard / design guide behavior -> redo using TP01_PROMPTS.md strict B rules.
AnchorC forms a complete story -> split into reusable asset vocabulary.
manifest too vague -> add routing, governance, A/B/C separation, runtime mapping, prompt module mapping, validation status.
only one image generated -> split into AnchorA/B/C.
only SHA was read -> report metadata-only state and stop before runtime generation.
overview image generated -> reject and regenerate A/B/C separately.
```


Rules:

```text
- If visual read is failed for any anchor, permission to continue must be no.
```

Must report:

```text
- AnchorA.runtime.md: created / failed

- canonical baseline leakage: yes / no
- prompt module available for generation: yes / no
- permission to continue to anchor image generation: yes / no
```

Must report:

```text
- AnchorA.PNG: generated / failed

- AnchorC.PNG: generated / failed
```
