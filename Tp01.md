# Tp01 Anchor Extraction Procedure

version: 1.3
mode: procedure
purpose: 將任意參考圖逐一提取成一組可重用的 Anchor Set

---

## 0. Output Contract

TA 根目錄必須有一份總控檔：

```text
TA/
  Tp01.md
  SEED_MASTER_INDEX.md
```

`SEED_MASTER_INDEX.md` 是全局總控，用來記錄每個 seed 是什麼、適合做什麼、目前狀態、Drive 位置、reference 摘要與 anchor 完成度。

每次執行 Tp01，必須輸出 4 個獨立檔案：

```text
AnchorA.PNG
AnchorB.PNG
AnchorC.PNG
manifest.md
```

嚴格規則：
- `AnchorA.PNG`、`AnchorB.PNG`、`AnchorC.PNG` 必須是三張不同圖卡。
- `manifest.md` 必須是單一 Markdown 檔。
- 不可把 A/B/C 合成一張圖。
- 不可只輸出一張總覽圖。
- 不可用 `manifest.md` 取代三張 Anchor 圖卡。
- 不可用三張原圖裁切直接當 Anchor。
- 每張 Anchor 只負責自己的單一職責。

資料夾完成狀態只接受：

```text
SEED XX | <用途型風格名稱>/
  SOURCE_DIR/
    SOURCE_INDEX.md
    REF_XX_01.<ext>
  AnchorA.PNG
  AnchorB.PNG
  AnchorC.PNG
  manifest.md
```

---

## 1. Master Index Contract

`SEED_MASTER_INDEX.md` 是整個 TA seed library 的總控檔。

目的：
- 記錄所有 seed 的用途與定位。
- 快速判斷哪個 seed 適合哪種任務。
- 記錄每個 seed 的 Drive folder。
- 記錄每個 seed 是否已有 A/B/C/manifest/source。
- 記錄主要 reference 摘要，但不取代每個 seed 的 `SOURCE_INDEX.md`。
- 每次 call `Tp01.md` 建立、修改、重做 seed 後都必須更新。

總控與個別 source 的分工：

```text
SEED_MASTER_INDEX.md
  = 全局總表：每個 seed 是什麼、用在哪裡、目前狀態

SEED XX/SOURCE_DIR/SOURCE_INDEX.md
  = 單一 seed 的 reference 明細：用了哪些 ref、分別抽取了什麼
```

嚴格規則：
- 每次新增 seed，必須新增一列到 `SEED_MASTER_INDEX.md`。
- 每次重做 seed，必須更新該 seed 的 status / updated_at / change_log。
- 每次停用 seed，必須在 master index 標記為 `archived` 或 `do_not_use`。
- 不可只更新單一 seed 的 `manifest.md` 而不更新 master index。
- 不可只靠資料夾名稱記憶 seed 用途。

---

## 2. SEED_MASTER_INDEX.md Schema

`SEED_MASTER_INDEX.md` 必須使用以下結構：

```md
# SEED_MASTER_INDEX

library: TA Anchor Seed Library
procedure: Tp01.md
last_updated: <YYYY-MM-DD>

## Seed Registry

| seed_id | seed_name | role | use_case | style_family | drive_folder | anchors | source | status | updated_at |
|---|---|---|---|---|---|---|---|---|---|
| SEEDXX | <name> | derived_seed | <what it is for> | <style family> | <Drive URL or folder id> | A/B/C/manifest | SOURCE_INDEX.md | active | <YYYY-MM-DD> |

## Routing Summary

### Best For
- SEEDXX: <when to use>

### Avoid For
- SEEDXX: <when not to use>

## Change Log

- <YYYY-MM-DD>: Added / updated SEEDXX. Reason: <reason>.
```

`status` 可使用：

```text
active
experimental
needs_review
archived
do_not_use
```

`anchors` 必須標記：

```text
missing
A only
A+B
A+B+C
A+B+C+manifest
needs_redo
```

## 3. Source Directory Contract

每個 seed 必須有一個 `SOURCE_DIR`，用來保存與管理此 seed 的所有 reference。

```text
SEED XX | <用途型風格名稱>/
  SOURCE_DIR/
    SOURCE_INDEX.md
    REF_XX_01.<ext>
    REF_XX_02.<ext>
    REF_XX_03.<ext>
```

`SOURCE_DIR` 的目的：
- 保存 reference image / screenshot / link / prompt / context。
- 記錄每個 reference 被用來抽取什麼。
- 防止日後重新提取 anchor 時失去來源脈絡。
- 讓 seed 可以被 audit、redo、extend。

嚴格規則：
- 每次 call `Tp01.md` 建立或重做 seed，都必須更新 `SOURCE_DIR/SOURCE_INDEX.md`。
- 如果 reference 是本地圖片，需複製或上傳到 `SOURCE_DIR`。
- 如果 reference 是 URL，需在 `SOURCE_INDEX.md` 記錄 URL 與擷取日期。
- 如果 reference 來自 ChatGPT / Drive / 截圖，需記錄來源位置與用途。
- 不可只把 reference 放在聊天記錄裡。
- 不可只在 `manifest.md` 記錄 reference；reference 管理必須集中在 `SOURCE_INDEX.md`。

---

## 4. SOURCE_INDEX.md Schema

每個 `SOURCE_DIR` 必須有一份 `SOURCE_INDEX.md`。

```md
# SOURCE_INDEX

seed_id: SEEDXX
seed_name: <用途型風格名稱>
source_dir: SOURCE_DIR
last_updated: <YYYY-MM-DD>

## Reference Registry

| ref_id | file_or_url | source_type | used_for | extraction_note | status |
|---|---|---|---|---|---|
| REF_XX_01 | REF_XX_01.png | image | A/B/C/manifest | <what is extracted> | active |

## Extraction Mapping

### AnchorA Layout References
- REF_XX_01: <layout grammar extracted>

### AnchorB Style References
- REF_XX_01: <style DNA extracted>

### AnchorC Asset References
- REF_XX_01: <asset vocabulary extracted>

### Manifest / Routing References
- REF_XX_01: <use-case positioning extracted>

## Update Log

- <YYYY-MM-DD>: created seed source registry.
- <YYYY-MM-DD>: updated after Tp01 call; changed <A/B/C/manifest>.
```

`used_for` 只可使用：

```text
A
B
C
manifest
A+B
A+C
B+C
A+B+C
A+B+C+manifest
```

`source_type` 可使用：

```text
image
screenshot
url
chatgpt_output
drive_file
prompt
note
```

`status` 可使用：

```text
active
archived
replaced
do_not_use
```

## 5. Source Update Protocol

每次 call `Tp01.md` 時，必須執行以下 source update：

1. 確認目前 seed folder 是否已有 `SOURCE_DIR`。
2. 若沒有，建立 `SOURCE_DIR`。
3. 確認 `SOURCE_DIR/SOURCE_INDEX.md` 是否存在。
4. 若沒有，建立 `SOURCE_INDEX.md`。
5. 將新的 reference 放入 `SOURCE_DIR`，並命名為：

```text
REF_XX_01.<ext>
REF_XX_02.<ext>
REF_XX_03.<ext>
```

6. 在 `SOURCE_INDEX.md` 新增或更新 reference registry。
7. 在 extraction mapping 中記錄該 reference 影響了哪個 Anchor。
8. 完成 `AnchorA/B/C/manifest` 後，在 update log 回寫本次處理結果。

如果只是重做某一張 Anchor：
- 不建立新 seed。
- 不重置 `SOURCE_INDEX.md`。
- 只新增一筆 update log。
- 若用了新 reference，新增新 `REF_XX_NN`。
- 若停用舊 reference，將舊 reference status 改為 `replaced` 或 `do_not_use`。

## 6. Core Contract

Reference image 只可作為抽取依據，不可直接複製。

允許抽取：
- layout grammar
- visual style DNA
- object / scene vocabulary
- use-case positioning

禁止保留：
- 原始角色身份
- 原始品牌名稱
- 原始完整文字
- 原圖特定主題結論
- 原圖完整構圖成品感
- 可能造成侵權的特定畫風或角色辨識特徵

若 reference image 風格粗糙、品牌化、或有侵權風險，必須只抽結構與功能，並轉寫成使用者常用的柔和日系手繪 anchor style。

## 7. Sequential Execution Lock

必須逐一製作，不可跳步。

執行順序固定如下：

```text
STEP 0 -> SEED_MASTER_INDEX.md
STEP 1 -> SOURCE_DIR / SOURCE_INDEX.md
STEP 2 -> AnchorA.PNG
STEP 3 -> AnchorB.PNG
STEP 4 -> AnchorC.PNG
STEP 5 -> manifest.md
STEP 6 -> update SOURCE_INDEX.md
STEP 7 -> update SEED_MASTER_INDEX.md
STEP 8 -> upload / verify
```

每一步完成標準：
- STEP 0 確認或建立全局 `SEED_MASTER_INDEX.md`。
- STEP 1 完成 source 登記後，才可開始 AnchorA。
- STEP 2 完成 AnchorA 後，才可開始 AnchorB。
- STEP 3 完成 AnchorB 後，才可開始 AnchorC。
- STEP 4 完成 AnchorC 後，才可開始 manifest。
- STEP 5 完成後，必須回寫 `SOURCE_INDEX.md`。
- STEP 6 完成後，必須回寫 `SEED_MASTER_INDEX.md`。
- STEP 8 完成後，才可結束。

如果使用者要求「先給我看」，只輸出當前 step，不提前製作後續 step。

## 8. Anchor Object Model

### STEP 2: AnchorA.layout_grammar()

輸出檔名：

```text
AnchorA.PNG
```

目的：只提取版面骨架。

必須包含：
- 頁面比例
- 標題區位置
- 主內容區位置
- 人物 / 圖像 placeholder 位置
- 補充資訊區位置
- 結論 / takeaway 區位置
- 流程、箭頭、panel、grid、card 的空間關係
- 留白比例與閱讀順序

視覺要求：
- blank skeleton
- grey placeholders
- layout only
- reusable
- 不放可讀內容
- 不放完整角色
- 不放完整物件

禁止包含：
- 完整人物
- 完整物件
- 完整可讀文字
- 原圖主題內容
- 角色風格
- 物件情緒
- 成品式插圖

判定失敗：
- 看起來像完成圖卡，而不是骨架。
- 有原圖文字或原圖主題。
- 人物或物件已經畫到可作為正式素材。

---

### STEP 3: AnchorB.style_dna()

輸出檔名：

```text
AnchorB.PNG
```

目的：只提取視覺風格 DNA。

必須包含：
- 線條質感
- 色票
- 字感
- icon 語言
- 人物比例
- 表情語法
- 邊框風格
- 裝飾元素
- 整體視覺語氣

視覺要求：
- style sheet
- visible style samples
- no fixed final composition
- no topic dependency
- 不複製 AnchorA 的 layout
- 不承擔完整資訊架構

禁止包含：
- 固定完整 layout
- 完整圖卡內容
- 原圖故事
- 物件清單
- 任務流程
- 主題結論

判定失敗：
- 看起來像一張完整資訊圖。
- 只是原圖換皮。
- 角色、文字、layout 混在一起，無法單獨作為 style DNA。

---

### STEP 4: AnchorC.asset_vocabulary()

輸出檔名：

```text
AnchorC.PNG
```

目的：只提取可重用素材語彙。

必須包含：
- 人物姿勢
- 表情
- 手勢
- 道具
- 場景元件
- icon
- 箭頭
- 卡片
- 標籤
- 裝飾符號
- 可重用 UI / infographic 元件

視覺要求：
- asset mini sheet
- object vocabulary
- scene vocabulary
- reusable parts only
- 不複製 AnchorA 的 layout
- 不承擔 AnchorB 的整體風格解釋

禁止包含：
- 完整頁面構圖
- 完整故事
- 完整版面 hierarchy
- 單一主題結論
- 原圖內容照搬

判定失敗：
- 看起來像完整圖卡。
- 素材之間形成固定故事。
- 物件直接保留原圖特定品牌或身份。

---

### STEP 5: manifest.write()

輸出檔名：

```text
manifest.md
```

目的：用單一 Markdown 檔記錄 seed 的用途、限制、路由規則與 A/B/C 職責。

必須包含：
- seed name
- role
- canonical status
- suitable_for
- not_suitable_for
- visual parameters
- anchor dependency
- routing rule
- governance rule
- A/B/C separation rule

禁止：
- 把三張 Anchor 圖卡內容塞進 manifest 取代圖片。
- 寫成普通說明文而沒有可執行規格。
- 只描述美感，不描述使用邊界。

## 9. Default Style Policy

除非使用者明確指定，所有新 seed 預設使用：
- 日系柔和手繪風
- 乾淨可讀
- 低飽和色
- 細線稿
- 白底或淡紙感背景
- 圓角模組
- 適合社群圖卡
- 不使用粗糙原圖直出風格

Default transformation:
- copy structure, not identity
- copy workflow logic, not brand
- copy visual function, not exact artwork
- copy mood category, not artist-specific style

---

## 10. Seed Naming Rule

新 seed 使用下一個編號：

```text
SEED XX | <用途型風格名稱>
```

風格名稱必須描述用途，不描述原圖主題或品牌。

正確：
- SEED 10 | 日系角色圖鑑物件收藏風
- SEED 11 | 柔和日系格線紙情緒獨白風
- SEED 12 | 柔和日系 SaaS 工作流圖卡

錯誤：
- SEED XX | Claude 教學圖
- SEED XX | 水晶女巫圖
- SEED XX | 某作者風格

## 11. Manifest Template

每個 seed 必須產生一份 `manifest.md`。

```md
# SEEDXX Manifest

role: derived_seed
name: <seed name>
canonical_status: derived_from_basal
default_ratio: <3:4 / 1:1 / other>
date_policy: forbidden

## Suitable For
- ...

## Not Suitable For
- ...

## Visual Parameters
- layout_density:
- character_load:
- mechanism_capacity:
- evidence_capacity:
- object_capacity:
- text_capacity:
- workflow_capacity:
- tone:

## Anchor Outputs
- AnchorA.PNG: layout grammar only
- AnchorB.PNG: style DNA only
- AnchorC.PNG: object / scene vocabulary only

## Anchor Use
- AnchorA: defines structure and reading order only.
- AnchorB: defines visual language only.
- AnchorC: defines reusable asset vocabulary only.

## Routing Rule
- Use this seed when ...
- Avoid this seed when ...

## Governance Rules
- Do not directly reproduce the reference image.
- Do not preserve specific brands, names, exact wording, or original identity.
- Use reference only as broad inspiration.
- Keep A/B/C separation strict.
- Output must remain three separate Anchor image cards plus one manifest.md.
- Source references must be registered in SOURCE_DIR/SOURCE_INDEX.md.
- Seed purpose and status must be registered in TA/SEED_MASTER_INDEX.md.
```

## 12. Full Execution Procedure

When user provides a reference image and asks to create a seed:

1. Inspect the reference image.
2. Identify seed type:
   - educational explainer
   - workflow infographic
   - character catalog
   - emotional monologue
   - poster card
   - comic infographic
   - scientific diagram
   - other
3. Assign next seed number.
4. Open or create TA root `SEED_MASTER_INDEX.md`.
5. Create seed folder locally.
6. Create or update `SOURCE_DIR`.
7. Create or update `SOURCE_DIR/SOURCE_INDEX.md`.
8. Register all reference sources before generating anchors.
9. Generate `AnchorA.PNG` only.
10. Validate `AnchorA.PNG`.
11. Generate `AnchorB.PNG` only.
12. Validate `AnchorB.PNG`.
13. Generate `AnchorC.PNG` only.
14. Validate `AnchorC.PNG`.
15. Write one `manifest.md`.
16. Update `SOURCE_DIR/SOURCE_INDEX.md` with extraction mapping and update log.
17. Update `SEED_MASTER_INDEX.md` with seed purpose, routing, status, folder link, and anchor completion.
18. Upload all required files to Google Drive target folder when requested.
19. Verify target folder contains exactly the required output files and `SOURCE_DIR`.

Required final folder check:

```text
TA/SEED_MASTER_INDEX.md exists
SOURCE_DIR exists
SOURCE_DIR/SOURCE_INDEX.md exists
AnchorA.PNG exists
AnchorB.PNG exists
AnchorC.PNG exists
manifest.md exists
No combined Anchor overview image is used as replacement
```

## 13. Validation Checklist

Before finalizing:

- TA root SEED_MASTER_INDEX.md exists.
- SEED_MASTER_INDEX.md has a row for this seed.
- SEED_MASTER_INDEX.md records use_case, style_family, status, and anchor completion.
- SOURCE_DIR exists.
- SOURCE_INDEX.md exists.
- Every active reference is registered in SOURCE_INDEX.md.
- SOURCE_INDEX.md records which reference was used for A/B/C/manifest.
- AnchorA is layout only.
- AnchorB is style only.
- AnchorC is asset vocabulary only.
- A/B/C are three separate image files.
- Manifest is one separate Markdown file.
- No original text is copied.
- No brand names remain unless explicitly requested.
- No full card is duplicated as A/B/C.
- Manifest describes use and limits clearly.
- Seed is distinct from existing seeds.
- Drive folder contains the expected Anchor package plus SOURCE_DIR.


## 15. Failure Handling

如果輸出不符合規格，必須指出是哪一步失敗，並只重做失敗的檔案。

常見修正：
- 沒有 SEED_MASTER_INDEX.md -> 在 TA root 補建總控檔。
- SEED_MASTER_INDEX.md 沒有該 seed -> 補登 seed 用途、狀態與 Drive folder。
- 沒有 SOURCE_DIR -> 補建 SOURCE_DIR。
- 沒有 SOURCE_INDEX.md -> 補建並登記所有 active reference。
- SOURCE_INDEX.md 沒有 update log -> 補回本次 Tp01 call 的處理記錄。
- AnchorA 太像完成圖卡 -> 重做為空白骨架。
- AnchorB 混入 layout -> 移除固定版面，只保留 style DNA。
- AnchorC 太像完整故事 -> 拆成可重用素材語彙。
- manifest 太鬆散 -> 補上 routing、governance、A/B/C separation。
- 只產生一張圖 -> 重新分拆為三張 Anchor 圖卡。
