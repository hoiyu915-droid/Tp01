## 0. Purpose

TP01 is the daily-use anchor extraction workflow for turning one or more reference images into a reusable Anchor Set.

This file is the executable procedure. It must be read from GitHub when the user invokes `TP01` / `tp01` without attaching a local TP01 file.

TP01 no longer uses Google Drive as the default source for the procedure file or seed anchors.

Version 2.1.0 adds three mandatory stop points so that source reading, anchor generation, and package assembly cannot be silently collapsed into one unchecked run.

## 1. Source Authority

Canonical source order:

```text
1. GitHub repo metadata
2. TP01_SOURCE_LOCK.md
3. Tp01.md
4. raw AnchorA.PNG / AnchorB.PNG / AnchorC.PNG
```

Required canonical raw links:

```text
Tp01.md
https://raw.githubusercontent.com/hoiyu915-droid/Tp01/main/Tp01.md

AnchorA.PNG
https://raw.githubusercontent.com/hoiyu915-droid/Tp01/main/AnchorA.PNG

AnchorB.PNG
https://raw.githubusercontent.com/hoiyu915-droid/Tp01/main/AnchorB.PNG

AnchorC.PNG
https://raw.githubusercontent.com/hoiyu915-droid/Tp01/main/AnchorC.PNG
```

The GitHub connector is used for:

```text
- repository metadata
- file SHA
- Tp01.md text
- TP01_SOURCE_LOCK.md text
- text/procedure audit
```

Raw GitHub links are used for:

```text
- AnchorA.PNG visual content
- AnchorB.PNG visual content
- AnchorC.PNG visual content
```

Important distinction:

```text
File existence / SHA / metadata is not visual read.
An anchor may be called "read" only after the actual image content has been opened or visually inspected.
```

## 2. No-GDrive Default Rule

When the user invokes `TP01` / `tp01` without attachments:

```text
Do not use GDrive as the default TP01 template source.
Do not use GDrive as the default seed anchor source.
Do not search Drive for the TP01 procedure.
Do not search Drive for AnchorA/B/C.
Do not silently fall back to Drive if GitHub raw images fail.
```

GDrive may only be used when the user explicitly says one of the following:

```text
upload to Drive
Drive audit
Drive verification
use GDrive TP01
use Drive anchors
check the Drive folder
```

If the user explicitly requests Drive, that instruction overrides this default.

## 4. Invocation Contract

When the user says `tp01` and does not attach a TP01 file:

1. Fetch repository metadata from `hoiyu915-droid/Tp01`.
2. Read `TP01_SOURCE_LOCK.md`.
3. Read `Tp01.md`, including SHA / metadata.
4. Read AnchorA / AnchorB / AnchorC from the canonical raw links.
5. Verify all four source components are available:
   - `Tp01.md`
   - `AnchorA.PNG`
   - `AnchorB.PNG`
   - `AnchorC.PNG`
6. **STOP POINT 1 — Anchor Read Report**
   - Before any generation, output a report confirming whether AnchorA / B / C were actually read as visual content.
   - This report must distinguish:
     - file existence / metadata
     - SHA availability
     - actual visual-read success
     - unresolved failures
   - If any anchor image was not actually read visually, stop the TP01 run here.
7. Inspect the user reference / task only after Stop Point 1 passes.
8. Generate `AnchorA.PNG`, `AnchorB.PNG`, and `AnchorC.PNG` sequentially.
9. **STOP POINT 2 — Generated Anchor Report**
   - After generating A / B / C, output a report evaluating:
     - whether AnchorA is layout grammar only
     - whether AnchorB is style DNA only
     - whether AnchorC is asset / object vocabulary only
     - whether the three anchors are sufficiently separated
     - whether any anchor drifted into a finished card or full story
   - Do not write `manifest.md`, finalize `SOURCE_INDEX.md`, create a zip, or upload before this report.
10. Write `manifest.md` and finalize `SOURCE_DIR/SOURCE_INDEX.md` only after Stop Point 2 passes.
11. **STOP POINT 3 — Package Preflight Report**
   - Before building the final seed package / zip, output a report confirming:
     - all required files exist
     - source references are recorded
     - manifest is aligned with A / B / C
     - no overview image replaced A / B / C
     - package is complete and ready
12. Generate zip / upload only after Stop Point 3 passes and only if the user requested a package / zip / upload.
13. Report any failed source explicitly.
14. Do not substitute GDrive unless the user explicitly asks for it.

Hard rule:

```text
Do not collapse TP01 into a single uninterrupted generate-and-package run.
The three stop points are mandatory execution gates.
```

## 5. Output Contract

Each TP01 extraction run produces a complete Anchor Package.

Required outputs:

```text
AnchorA.PNG
AnchorB.PNG
AnchorC.PNG
manifest.md
```

Strict rules:

```text
- AnchorA.PNG, AnchorB.PNG, and AnchorC.PNG must be three separate image files.
- Do not combine A/B/C into one overview image.
- Do not replace the three images with manifest.md.
- Do not crop the original reference directly and call it an anchor.
- Each anchor has one job only.
- Generate complete files directly; do not output patch-only files.
```

Optional local package structure:

```text
TP01_OUTPUT/
  SOURCE_DIR/
    SOURCE_INDEX.md
    REF_01.<ext>
    REF_02.<ext>
  AnchorA.PNG
  AnchorB.PNG
  AnchorC.PNG
  manifest.md
```

If the user asks for a zipped package, produce a complete zip containing the full folder, not partial patches.

## 6. Anchor Separation Model

### AnchorA.PNG — layout grammar only

Purpose:

```text
Extract page skeleton, reading order, spacing, panels, card hierarchy, and content zones.
```

Must include:

```text
- page ratio
- title zone
- main content zone
- visual / character placeholder zone
- supporting note zone
- conclusion / takeaway zone
- arrows, panels, grid, card relationship
- whitespace ratio
- reading order
```

Must not include:

```text
- complete character
- complete object
- readable final text
- original topic content
- finished illustration
- brand / identity carryover
```

Failure condition:

```text
If AnchorA looks like a finished infographic rather than a reusable skeleton, it fails.
```

---

### AnchorB.PNG — style DNA only

Purpose:

```text
Extract visual language without fixing the final layout.
```

Must include:

```text
- line quality
- palette
- typography feel
- icon language
- character proportion
- expression grammar
- border style
- decoration style
- overall visual tone
```

Must not include:

```text
- fixed full layout
- complete infographic content
- original story
- object inventory
- workflow logic
- topic conclusion
```

Failure condition:

```text
If AnchorB becomes a full information card or simply a recolored copy of the reference, it fails.
```

---

### AnchorC.PNG — asset / object vocabulary only

Purpose:

```text
Extract reusable visual components and object vocabulary.
```

Must include:

```text
- poses
- expressions
- gestures
- props
- scene elements
- icons
- arrows
- labels
- cards
- reusable UI / infographic components
```

Must not include:

```text
- complete page composition
- complete story
- full hierarchy
- single fixed conclusion
- brand-specific or identity-specific objects copied from the source
```

Failure condition:

```text
If AnchorC reads like a finished story scene instead of a reusable asset sheet, it fails.
```

## 9. Default Style Policy

Unless the user explicitly gives another style, new TP01 anchor packages should use:

```text
- portrait / vertical layout by default
- soft Japanese hand-drawn card style
- clean readable structure
- low-saturation palette
- fine linework
- white or pale paper background
- rounded modules
- social-card readability
```

Default transformation rule:

```text
copy structure, not identity
copy workflow logic, not brand
copy visual function, not exact artwork
copy mood category, not artist-specific style
```

## 10. Execution Order

TP01 must be executed sequentially:

```text
STEP 0  -> read TP01_SOURCE_LOCK.md
STEP 1  -> read Tp01.md / SHA / metadata
STEP 2  -> read raw AnchorA.PNG
STEP 3  -> read raw AnchorB.PNG
STEP 4  -> read raw AnchorC.PNG
STEP 5  -> STOP POINT 1: Anchor Read Report
STEP 6  -> inspect user reference / task
STEP 7  -> create SOURCE_DIR / SOURCE_INDEX.md draft if creating a package
STEP 8  -> generate AnchorA.PNG
STEP 9  -> validate AnchorA.PNG
STEP 10 -> generate AnchorB.PNG
STEP 11 -> validate AnchorB.PNG
STEP 12 -> generate AnchorC.PNG
STEP 13 -> validate AnchorC.PNG
STEP 14 -> STOP POINT 2: Generated Anchor Report
STEP 15 -> finalize SOURCE_INDEX.md
STEP 16 -> write manifest.md
STEP 17 -> STOP POINT 3: Package Preflight Report
STEP 18 -> final package validation
STEP 19 -> zip or upload only if explicitly requested
```

Do not skip from reference image directly to final card.

Do not make AnchorB before AnchorA.

Do not make AnchorC before AnchorB.

Do not write manifest before A/B/C exist, unless the user explicitly asks for a planning-only manifest.

Do not zip or upload before Stop Point 3 passes.

Strict gate rule:

```text
Do not skip Stop Point 1.
Do not generate anchors before the Anchor Read Report.
Do not generate the seed package before the Generated Anchor Report.
Do not zip or upload before the Package Preflight Report.
```

## 12. Failure Handling

If any required GitHub source fails:

```text
- Say exactly which file failed.
- Distinguish metadata success from visual-read success.
- Do not claim an anchor was "read" if only SHA / existence was confirmed.
- Do not infer anchor visual content from Tp01.md text alone.
- Do not infer anchor visual content from memory, prior conversation, or previous seed packages.
- Do not silently search GDrive.
- Ask for explicit override only if the user wants to continue with another source.
- Stop before generation if Stop Point 1 cannot pass.
```

If generated anchors fail separation rules:

```text
- Stop after Stop Point 2.
- Report which anchor failed and why.
- Regenerate the failed anchor only.
- Do not write manifest or build package until AnchorA / B / C all pass.
```

If package assembly fails:

```text
- Stop at Stop Point 3.
- Report missing or inconsistent files.
- Do not zip incomplete outputs.
```

Common failures and corrections:

```text
Tp01.md still mentions Drive as default -> rewrite procedure to GitHub canonical source.
AnchorA too finished -> redo as layout skeleton.
AnchorB includes full layout -> remove fixed composition.
AnchorC forms a complete story -> split into reusable asset vocabulary.
manifest too vague -> add routing, governance, A/B/C separation, validation status.
only one image generated -> split into AnchorA/B/C.
only SHA was read -> report metadata-only state and stop before generation.
overview image generated -> reject and regenerate A/B/C separately.
```

## 13. Stop Point Report Contract

### STOP POINT 1 — Anchor Read Report

Must report:

```text
- repository:
- Tp01.md sha:
- AnchorA sha:
- AnchorB sha:
- AnchorC sha:
- AnchorA metadata/existence: success / failed
- AnchorB metadata/existence: success / failed
- AnchorC metadata/existence: success / failed
- AnchorA visual read: success / failed
- AnchorB visual read: success / failed
- AnchorC visual read: success / failed
- unresolved gaps:
- permission to continue: yes / no
```

Rules:

```text
- If visual read is failed for any anchor, permission to continue must be no.
- Do not generate anchors from memory, old seed, or role definitions.
- Do not treat Tp01.md role descriptions as proof of actual anchor image content.
```

### STOP POINT 2 — Generated Anchor Report

Must report:

```text
- AnchorA status: pass / fail
- AnchorB status: pass / fail
- AnchorC status: pass / fail
- separation integrity: pass / fail
- notable drift:
- fixes required:
- permission to continue to manifest/package: yes / no
```

Rules:

```text
- If any anchor fails, permission to continue must be no.
- Do not generate manifest or package before this report passes.
```

### STOP POINT 3 — Package Preflight Report

Must report:

```text
- files present:
- SOURCE_INDEX.md status:
- manifest.md status:
- anchor-manifest alignment:
- packaging completeness:
- zip readiness: yes / no
```

Rules:

```text
- If any required file is missing or inconsistent, zip readiness must be no.
- Do not create a zip or upload incomplete outputs.
```
